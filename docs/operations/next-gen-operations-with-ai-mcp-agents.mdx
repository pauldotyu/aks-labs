Imagine managing production clusters using natural language-asking agents to diagnose issues, optimize configurations, and execute operations without memorizing complex CLI syntax.

In this workshop, you will experience the future of Azure Kubernetes Service (AKS) operations through AI agents.

You will explore two complementary approaches:

- **[CLI Agent for AKS](https://github.com/Azure/cli-agent-for-aks)**: Terminal-based AI assistant for interactive cluster management
- **[kagent](https://kagent.dev/)**: A CNCF Sandbox project that brings AI agents directly into your Kubernetes cluster as native resources

The common thread between these to approaches being the **[AKS MCP Server](https://github.com/Azure/aks-mcp)**, a standardized interface that exposes Azure, AKS, and Kubernetes capabilities to AI agents.

Together, these tools demonstrate how AI can transform cluster operations-from troubleshooting network connectivity to managing multi-cluster deployments-all through intelligent, context-aware automation.

### Lab roadmap

This lab is organized into progressive phases that guide new and experienced administrators from foundational setup to advanced production operations:

1. **Environment preparation**: Authenticate to Azure, GitHub Enterprise, and connecting to the AKS cluster.
2. **Tool familiarization**: Explore the AKS MCP server, the CLI Agent for AKS, and kagent so you understand how each agent contributes to the overall workflow.
3. **Traffic spike triage**: Generate load, observe AI-driven alerts, and guide the agents through an automated scale plan that keeps services stable.
4. **Node remediation and patch governance**: Simulate unhealthy nodes, watch the agents orchestrate self-healing actions, and confirm image compliance.
5. **Proactive cluster health assessment**: Use multi-agent collaboration to analyze cluster health, implement Azure Advisor recommendations, and create executive-ready optimization roadmap.

Each phase builds on the previous one.

### What you will learn

By the end of this lab, you will be able to:

- **Deploy and configure** AI agent frameworks (AKS MCP Server, CLI Agent, kagent) with Azure AI Foundry models
- **Build multi-agent systems** where specialized agents collaborate on complex operational tasks
- **Troubleshoot production scenarios** using natural language instead of memorizing CLI syntax
- **Orchestrate automated remediation** for node failures, traffic spikes, and configuration drift
- **Generate proactive health assessments** combining Azure Advisor recommendations with Kubernetes best practices
- **Understand the Model Context Protocol (MCP)** and how it standardizes AI-tool integration

## Introducing the Model Context Protocol (MCP)

The **[Model Context Protocol (MCP)](https://modelcontextprotocol.io/docs/getting-started/intro)** is an open standard that enables AI agents to access external tools and data sources in a consistent way. Developed by [Anthropic](https://www.anthropic.com/), MCP acts as a universal adapter between AI models and systems like databases, APIs, and cloud services.

### Key concepts

At a high level, here is what you need to know about MCP architecture.

- **MCP Servers** - Expose capabilities (tools, prompts, resources) through the protocol, (e.g., AKS MCP server).
- **MCP Clients** - Applications like AI agents that connect to MCP servers (e.g., GitHub Copilot Chat).
- **Tools** - Functions the AI calls to perform actions (kubectl commands, Azure APIs)
- **Resources** - Data sources providing context (cluster configs, logs)

:::info

If you are getting started with MCP, be sure to check out the [MCP for Beginners](https://github.com/microsoft/mcp-for-beginners) repo for an in-depth and step-by-step introduction.

:::

### The AKS MCP Server

The **AKS MCP Server** gives AI agents deep integration with Azure and Kubernetes, exposing tools for cluster management, diagnostics, and troubleshooting. Any MCP-compatible agent can use this server to perform AKS operations.

You will use the AKS MCP Server three ways:

1. **Locally with VS Code and [GitHub Copilot Agent](https://docs.github.com/en/copilot/concepts/agents/coding-agent/mcp-and-coding-agent)** - Test capabilities directly in VS Code
2. **Locally with CLI Agent for AKS** - Terminal-based operations
3. **In-cluster with kagent** - Multi-agent collaboration

## CLI Agent for AKS

The CLI Agent for AKS is an AI-powered command-line tool designed to help AKS users troubleshoot cluster issues efficiently.

:::note

The CLI Agent for AKS is currently in public preview and is still undergoing active development. Some features may change in future releases and your feedback is valuable to help improve the experience. Be sure to visit the [GitHub repository](https://github.com/Azure/cli-agent-for-aks) to report issues or suggest features.

:::

It analyzes telemetry signals (logs, metrics, events), correlates them across infrastructure and workloads, and provides actionable insights through natural language queries. 

The agent integrates with all the tools you'll need to operate AKS like kubectl, Azure CLI, Inspektor Gadget, and Azure Monitor, operating in read-only mode by default to ensure safe diagnostics. 

While the agent excels at surfacing likely causes and guiding investigation, human oversight is essential, particularly in complex or high-stakes environments.

### How it works

The agent is built on top of open-source project [HolmesGPT](https://holmesgpt.dev/), a CNCF sandbox project, and is packaged as an Azure CLI extension (**aks-agent**). This extension has been pre-installed in your lab environment. It will connect to Azure AI Foundry models to power its natural language understanding and uses the AKS MCP tools to interact with your AKS cluster.

### Configure the AI model

Before using the agent, you need to configure the backend language model. Azure AI Foundry provides the AI capabilities that enable the agent to understand your questions and provide intelligent responses. As mentioned above, the service and models has been pre-provisioned for you.

In VS Code, open a new terminal then run the following commands to create a the agent configuration file to use Azure AI Foundry models. These configuration will tell the agent where to find the AI service and how to authenticate to it.

```bash
cat << EOF > /home/labuser/.azure/aksAgent.yaml
model: azure/gpt-5-mini
llms:
- provider: azure
  DEPLOYMENT_NAME: gpt-5-mini
  AZURE_API_BASE: REPLACE_WITH_ACTUAL_VALUE
  AZURE_API_KEY: REPLACE_WITH_ACTUAL_VALUE
  AZURE_API_VERSION: 2024-12-01-preview
EOF
export TEMPERATURE=1
```

:::note

You can configure other models using the `az aks agent-config` command. See the [Install and Use the Agentic CLI for AKS documenation](https://learn.microsoft.com/azure/aks/cli-agent-for-aks-install?tabs=other-providers) for more information.

:::

:::warning

The **TEMPERATURE** setting controls how creative or predictable the AI responses are. The **gpt-5** family of models are highly-capable reasoning models that only supports temperature value of 1.

:::

Now start the agent session using the **gpt-5-mini** model which has also been pre-provisioned. This model is optimized for performance and cost-effectiveness while still providing intelligent responses.

```bash
az aks agent
```

This will drop you into an interactive terminal UI (TUI) session.

## kagent with Azure

**kagent** is a [Cloud Native Computing Foundation (CNCF) Sandbox project](https://www.cncf.io/sandbox-projects/) that brings AI agents directly into your Kubernetes cluster. Unlike the CLI agent which runs on your local machine, kagent agents run as Kubernetes resources within your cluster and can interact with your workloads in real-time. See the [kagent Architecture](https://kagent.dev/docs/kagent/concepts/architecture) for more details.

:::note

kagent is still in early development and is currently a CNCF Sandbox project. It is not yet production-ready and may have breaking changes in future releases. Follow the [kagent-dev/kagent](https://github.com/kagent-dev/kagent) repo for the latest development on the project.

:::

### Why kagent?

With the plethora of AI assisted tools, some may choose to use kagent for the following reasons:

- **Kubernetes-native** - Agents are defined as custom resources and managed with kubectl
- **Multi-agent architecture** - Create specialized agents that collaborate on complex tasks
- **CLI and Web UI** - Interact with agents through terminal or browser
- **Extensible** - Connect agents to MCP servers for additional capabilities

### Install kagent

kagent provides a CLI to easily install kagent into your clusters; however, you do need to customize the deployment to use Azure AI Foundry models, which requires using Helm charts. This gives us more control over the configuration.

To save you time and have you focus on using kagent rather than installing agent, the setup script you ran earlier included the kagent install command. Take a look at the installation notes below if you want to see exactly how to install it.

:::tip[kagent installation notes]

You do not need to run the installation commands below, but they are provided here to show you what the deployment process looks like if you want to install kagent in your own environments later.

The installation consists of two steps:

**Step 1: Install CRDs**

```bash
helm install kagent-crds oci://ghcr.io/kagent-dev/kagent/helm/kagent-crds \
--namespace kagent \
--create-namespace
```

**Step 2: Install kagent controller and UI**

```bash
helm install kagent oci://ghcr.io/kagent-dev/kagent/helm/kagent \
--namespace kagent \
--set providers.default=azureOpenAI \
--set providers.azureOpenAI.model="gpt-4.1-mini" \
--set providers.azureOpenAI.apiKey="REPLACE_WITH_ACTUAL_VALUE" \
--set providers.azureOpenAI.config.apiVersion="2024-12-01-preview" \
--set providers.azureOpenAI.config.azureDeployment="gpt-4.1-mini" \
--set providers.azureOpenAI.config.azureEndpoint="REPLACE_WITH_ACTUAL_VALUE"
```

:::

The installation includes:

- **kagent controller** - Manages agent lifecycles and reconciles custom resources
- **Dashboard UI** - A web interface to interact with agents
- **Default model configuration** - Initial OpenAI model config (you will replace this with Azure AI Foundry)
- **Pre-configured agents** - Specialized agents for Kubernetes, Istio, and Cilium operations

kagent uses Kubernetes Custom Resource Definitions (CRDs) to represent agents, models, and tools as native Kubernetes objects. This means you can manage agents using standard kubectl commands.

With the installation complete, you can now access the kagent dashboard.

In VS Code, open a new terminal and run the following command to get the LoadBalancer's public IP address of the kagent dashboard:

```bash
echo "http://$(kubectl get service kagent-ui -n kagent -o jsonpath='{.status.loadBalancer.ingress[0].ip}'):8080"
```

:::warning[Why load balancer?]

You might have noticed the type for the kagent-ui service is set to **LoadBalancer**. By default the kagent-ui dashboard is only available within the cluster. You could keep the service as-is and port-forward the service to access it locally, but occasionally the port-forwarded connection can be unstable causing you to lose your session. Using a LoadBalancer service type is more reliable just for this lab. In a production scenario, you should NOT expose this tool to the public internet.

:::

## AKS MCP with kagent

Earlier in the lab, you tested the AKS MCP server locally using GitHub Copilot. This same MCP server binary has been deployed into your cluster during the setup phase at the beginning of this lab. This server runs as a Kubernetes workload and provides AI agents with access to AKS-specific tools and diagnostics.

:::tip[AKS MCP server installation notes]

The AKS MCP server can take a few minutes to deploy so has also been pre-deployed for you to save time.

You do not need to apply the manifest below, but it is provided here to show you what the deployment includes if you want to deploy the AKS MCP server in your own environments later.

```yaml
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: aks-mcp
  namespace: kagent
  annotations:
    azure.workload.identity/client-id: REPLACE_WITH_ACTUAL_VALUE
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: aks-mcp-cluster-admin
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cluster-admin
subjects:
  - kind: ServiceAccount
    name: aks-mcp
    namespace: kagent
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: aks-mcp
  namespace: kagent
spec:
  replicas: 1
  selector:
    matchLabels:
      app: aks-mcp
  template:
    metadata:
      labels:
        app: aks-mcp
        azure.workload.identity/use: "true"
    spec:
      serviceAccountName: aks-mcp
      containers:
      - name: aks-mcp
        image: ghcr.io/azure/aks-mcp:v0.0.9
        args:
          - --access-level=readwrite
          - --transport=streamable-http
          - --host=0.0.0.0
          - --port=8000
          - --timeout=600
        ports:
        - containerPort: 8000
          name: http
        resources: {}
---
apiVersion: v1
kind: Service
metadata:
  name: aks-mcp
  namespace: kagent
spec:
  type: ClusterIP
  ports:
  - port: 8000
    targetPort: 8000
    protocol: TCP
    name: http
  selector:
    app: aks-mcp
```

:::

The deployment consists of four Kubernetes resources that were created in the **kagent** namespace:

1. **ServiceAccount (aks-mcp)** - Provides identity for the MCP server pod, linked to an Azure managed identity via annotations
2. **ClusterRoleBinding (aks-mcp-cluster-admin)** - Grants the service account cluster-admin permissions to perform diagnostic and management operations
3. **Deployment (aks-mcp)** - Runs the AKS MCP server container (ghcr.io/azure/aks-mcp:v0.0.9) with streamable HTTP transport
4. **Service (aks-mcp)** - Exposes the MCP server internally on port 8000 so agents can connect to it

:::info

**Azure Workload Identity** enables the MCP server to authenticate with Azure services without storing credentials. The service account has an annotation that links it to the Azure managed identity, which has the necessary permissions to manage AKS resources and query Azure APIs.

:::
